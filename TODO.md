Phase 0 — Branch & snapshot (15 min)
Cut ui/svelte-spa-migration off main. Keep the Astro tree intact but rename the directory to pidx-ui-astro/ so the new tree can live at pidx-ui/ without path collisions — this is a cosmetic move, not a deletion, and it means your tauri.conf.json identifiers, icon paths, and the pidx = { path = "../.." } workspace link don't need to change once you swap contents. Don't commit yet; workflows fire on push, and you want to land this as one coherent change. Do a quick cargo tauri dev pass on the Astro tree to confirm it still boots from the renamed path — catches any hardcoded pidx-ui/ references in build scripts before they bite.
Phase 1 — Scaffold SvelteKit SPA, prove one round-trip (1–2 hrs)
npm create svelte@latest pidx-ui with TypeScript, ESLint, Prettier. Swap the default adapter-auto for @sveltejs/adapter-static with fallback: 'index.html' and export const ssr = false in the root layout — that gives you true SPA mode, which is what Tauri wants. Point tauri.conf.json's frontendDist at pidx-ui/build and beforeDevCommand at the SvelteKit dev server (same port 1420, strict). Prove the round-trip with list_users only: one page, one invoke, rendered list. Do not port any other commands until this works end-to-end including hot reload. This is your go/no-go checkpoint — if something fundamental about Tauri v2 + SvelteKit + static disagrees, you want to know now with one command wired, not twelve.
Phase 2 — Typed IPC + design foundation (half day)
Add specta = { version = "2", features = ["derive"] } and tauri-specta = { version = "2", features = ["typescript"] } to pidx-ui/src-tauri/Cargo.toml, decorate your 13 commands with #[specta::specta], and emit a bindings.ts during dev. From the frontend you now import typed wrappers instead of calling invoke('confirm_observation', ...) with a stringly-typed name. This one change is the highest-leverage thing in the whole migration because every subsequent view becomes a refactor of typed data rather than a guess-and-check against IPC shapes.
In parallel, install Tailwind v4 (the Vite plugin flow, not the old PostCSS config) and pick one of Bits UI or Melt UI — I'd lean Bits UI, it's less ceremony and has better defaults for the dialog/toast primitives you'll need for the delta resolver and destructive confirm flows. Define design tokens (colors, spacing, font scale) in a single tokens.css so when you inevitably want to rework visual hierarchy between Gardener and Debugger modes you're editing one file.
Set your routing shape now so you don't retrofit it: / (Dashboard), /profile/[user] with nested (gardener), (inspector), (debugger) route groups sharing a layout, /bridge for the timeline, /diff for the matrix. SvelteKit's route groups give you persona-specific layouts without URL clutter.
Phase 3 — Port the Debugger first (1 day)
Debugger mode is the lowest-risk port because it's essentially a table over ProfileDocument — filters by status, sort by confidence or age, inline actions for clear <index> and reject-all <prefix>. It's also the mode you actually use day-to-day, so getting it working unblocks dogfooding the rest. Build the ObservationTable component generically enough that Inspector mode reuses it read-only. This is where you'll feel whether Svelte 5 runes are working for you — the filter/sort state is a textbook $state + $derived pattern.
Phase 4 — Inspector + Register viz (1 day)
Port get_show with tier toggle (T1/T2/T3/T4 segmented control). Build the register radar as its own component — six axes, recharts-equivalent in Svelte is either layerchart or plain D3 inside a Svelte component; for six axes D3 is fine, don't pull a library. Evidence drill-down opens a side panel, not a modal — modals are for decisions (delta resolver), panels are for inspection. Annotation overlay goes here: the T3 narrative block renders with annotation markers pinned to the observations they reference.
Phase 5 — Gardener loop (1–2 days)
This is where the UX spend pays off. Review queue and delta queue as card-feed inboxes, keyboard-first (J/K navigate, Y/N confirm/reject) with swipe gestures as an enhancement on touch. The delta resolver is its own flow — not a row in a table, a dedicated A/B screen with the full observation context on both sides. Empty states matter here; a cleared queue should feel like an accomplishment, not a blank page. If you're porting one persona per sitting, do this one last so the first-time-open experience is polished.
Phase 6 — UI-native features that earn their keep (ongoing)
In priority order: bridge packet authoring (drag text → tag field paths → emit .bridge.json), decay simulator (timeline scrub, all client-side math over the existing decay curve), bridge replay / undo from the timeline, tray + file watcher (Tauri v2 tray API wrapping the watch command with native notifications). None of these block each other. Pick one per session.
Phase 7 — ProfileMatrix primitive (1 day once you want it)
Build it generic from day one: { profiles: ProfileDocument[], fields: DotPath[], mode: 'confidence' | 'value' | 'age' }. v0 use case is temporal diff — snapshot the same profile at two timestamps (you'll need a snapshot command on the Rust side, or store historical saves) and feed both in. Source-stratified and tier-stratified views are the same component with different filtering upstream. When MIIN NPCs land on main, NPC-vs-self diff is a new call site, not a new component.
Phase 8 — Commit & PR
One commit per phase boundary is probably the right granularity, but since you're staging workflows, I'd actually rebase-squash Phases 0–2 into a single "migrate frontend to SvelteKit SPA + typed IPC" commit before pushing — reviewers don't care about the scaffolding steps, they care that the wiring changed atomically. Phases 3–5 can each be their own commit/PR since they're persona-scoped. Phase 6+ should each be independent PRs.
Schema-sync interaction points
Two moments where your miin-v0 pull could collide with this work: anything touching ProfileDocument struct shape invalidates the specta bindings — but that's a one-command regeneration, not a refactor. Anything adding new CLI subcommands should also get a Tauri command shim in commands.rs so the UI reaches parity in the same PR. If your pull adds new domain variants or origination codes, grep the frontend for any hardcoded enum string literals (there shouldn't be any if specta is generating them, but it's a 30-second check).
